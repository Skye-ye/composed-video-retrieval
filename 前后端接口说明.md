# 组合视频检索系统：前后端开发说明

> 面向：负责小组项目前端和后端开发的同学
> 模型侧：AIM + FDCA，已完成训练，对外只暴露三个 Python 函数
> 形态：固定视频库（用户不能上传），所有查询为"从库中选择参考视频 + 自然语言修改文本"

---

## 1. 总体架构

```
┌───────────┐   JSON   ┌───────────┐   Python 调用   ┌──────────────┐
│  前端 UI  │ ───────► │  后端服务  │ ──────────────► │  模型推理层   │
│ (Web 页面) │ ◄─────── │ (HTTP API) │ ◄────────────── │  AIM + FDCA  │
└───────────┘          └───────────┘                 └──────────────┘
                             │
                             ▼
                       ┌─────────────┐
                       │ Faiss 索引   │  ← 离线脚本预先写入
                       │ 视频元数据表  │     （由 AI 团队负责）
                       └─────────────┘
```

**关键边界：**

- 用户**不能上传视频**。所有可被检索的视频都来自一个**固定预置的视频库**（gallery），由 AI 团队离线完成抽帧、AIM 编码和 Faiss 索引构建。
- 每个视频对应**一个固定维度 D 的向量**，同一向量同时承担两种角色：作为候选库被 Faiss 检索、作为参考视频输入到 FDCA 融合层。
- 在线查询路径**不涉及任何视频文件传输、抽帧或重编码**，请求体始终是小体积 JSON。

## 2. 后端职责

### 2.1 启动时加载

后端进程启动时需要把以下三类预计算好的数据加载到内存：

| 资源 | 来源 | 形态 | 用途 |
|---|---|---|---|
| Faiss 索引文件 | AI 团队产出，例如 `index.faiss` | `IndexFlatIP`（内积相似度，向量已 L2 归一化） | Top-K 检索 |
| 视频 ID 映射表 | AI 团队产出，例如 `id_map.json` | `video_id (str) ↔ faiss 内部整型 id` | 在两侧 id 之间转换 |
| 视频元数据表 | AI 团队产出，例如 `metadata.json` 或 SQLite | `video_id → {thumbnail_url, duration_sec, title?, ...}` | 给前端返回展示信息 |

启动后所有数据常驻内存，不要每次请求重新读盘。

### 2.2 对外 HTTP 接口

只需要两个接口：

#### `GET /api/videos`

返回视频库列表，供前端在"选择参考视频"页面展示。支持游标分页。

**请求参数：**
- `cursor`（可选，string）：上一页返回的 `next_cursor`，第一页不传
- `limit`（可选，int，默认 24）：每页数量

**返回：**
```json
{
  "videos": [
    {
      "video_id": "vid_000123",
      "thumbnail_url": "/static/thumbs/vid_000123.jpg",
      "duration_sec": 12.4,
      "title": "可选字段"
    }
  ],
  "next_cursor": "eyJvZmZzZXQiOjI0fQ=="
}
```

#### `POST /api/search`

执行一次组合视频检索。

**请求体（application/json）：**
```json
{
  "reference_video_id": "vid_000123",
  "modification_text": "把跑步换成走路",
  "retain_text": null,
  "exclude_text": null,
  "top_k": 20,
  "debug": false
}
```

字段说明：
- `reference_video_id`（必填）：必须存在于视频库；不存在则返回 `404`
- `modification_text`（必填，非空字符串）
- `retain_text` / `exclude_text`（可选）：仅在前端"高级模式"下提供，对应 FDCA 的保留 / 排除分支
- `top_k`（可选，默认 20，上限建议 50）
- `debug`（可选，默认 false）：是否返回模型调试信息

**返回：**
```json
{
  "query_id": "q_8f2a1c",
  "results": [
    {
      "video_id": "vid_000456",
      "score": 0.84,
      "thumbnail_url": "/static/thumbs/vid_000456.jpg",
      "duration_sec": 8.7,
      "low_confidence": false,
      "debug": {
        "branch_scores": {"retain": 0.71, "inject": 0.62, "exclude": 0.12},
        "predicted_action_class": "walk"
      }
    }
  ]
}
```

- `debug` 字段仅当请求中 `debug=true` 时出现
- `low_confidence`：当该结果的 `score` 低于约定阈值时为 `true`，由前端用于视觉降权（见 §3.4）

### 2.3 与模型层的对接

模型团队提供三个 Python 函数，全部无状态、可重入：

```python
def encode_query(
    ref_embedding: Tensor[D],            # 后端从 Faiss reconstruct 得到
    modification_text: str,
    retain_text: str | None = None,
    exclude_text: str | None = None,
    return_debug: bool = False,
) -> tuple[Tensor[D], dict | None]

def search(query: Tensor[D], top_k: int) -> list[tuple[str, float]]

# 仅离线索引脚本调用，后端服务无需关心
def encode_video(frames: Tensor[T,3,H,W]) -> Tensor[D]
```

**后端 `/api/search` 处理流程示意：**

```python
def handle_search(req):
    if req.reference_video_id not in id_map:
        raise HTTPException(404, "reference video not found")

    internal_id = id_map[req.reference_video_id]
    ref_vec = faiss_index.reconstruct(internal_id)            # Tensor[D]

    query_vec, debug = model.encode_query(
        ref_vec,
        req.modification_text,
        retain_text=req.retain_text,
        exclude_text=req.exclude_text,
        return_debug=req.debug,
    )

    # 多取一个，方便过滤参考视频自身
    hits = model.search(query_vec, top_k=req.top_k + 1)
    hits = [(vid, s) for vid, s in hits if vid != req.reference_video_id][:req.top_k]

    return build_response(hits, debug=debug if req.debug else None)
```

### 2.4 业务规则（必须实现）

1. **过滤参考视频本身**：检索结果中如果出现请求里的 `reference_video_id`，必须由后端剔除。原因是同一向量同时承担候选与参考角色，模型几乎一定会把参考视频排在 Top-1。
2. **低置信度标记**：对每个结果计算 `low_confidence = (score < THRESHOLD)`。`THRESHOLD` 在配置文件中定义，初值可取 `0.3`，后续根据评测调整。**不要直接丢弃低分结果**，标记后交由前端处理。
3. **请求校验**：`modification_text` 去除首尾空白后必须非空；`top_k` 限制在 `[1, 50]`。
4. **错误响应**统一为：
   
   ```json
   { "error": { "code": "INVALID_INPUT", "message": "modification_text is empty" } }
   ```
5. **延迟预算**：单次 `/api/search` 服务端处理目标 < 500ms（包含模型推理 + Faiss 检索）。如果稳定超过该值，先排查是否在请求路径里误用了 `encode_video` 或重新加载了模型。

### 2.5 不需要做的事

- ❌ 视频文件上传、转码、抽帧
- ❌ 用户账号、登录、鉴权（课程项目演示无需）
- ❌ 检索结果的持久化历史
- ❌ 重新计算或更新 Faiss 索引（这是 AI 团队的离线脚本职责）

## 3. 前端职责

### 3.1 页面结构

整个应用只有一个核心页面，由三块区域组成：

```
┌─────────────────────────────────────────────────────────┐
│  [顶部标题栏]  基于深度学习的组合视频检索                  │
├──────────────────────┬──────────────────────────────────┤
│                      │                                  │
│  ① 参考视频选择区     │  ③ 检索结果区                     │
│  （从视频库浏览/选中） │  （结果卡片网格，按相似度排序）    │
│                      │                                  │
│                      │                                  │
├──────────────────────┴──────────────────────────────────┤
│  ② 查询输入区                                            │
│  [修改文本输入框]                          [检索按钮]    │
│  ▾ 高级模式（折叠默认隐藏）                              │
│     [保留：什么内容应保持不变]                            │
│     [排除：什么内容不应出现]                              │
└─────────────────────────────────────────────────────────┘
```

### 3.2 核心交互流程

1. 用户进入页面，前端调用 `GET /api/videos` 拉取视频库第一页，渲染为缩略图网格。
2. 用户点击其中一个视频缩略图作为参考视频（高亮选中）。
3. 用户在修改文本框输入指令，可选展开"高级模式"补充保留/排除内容。
4. 点击检索按钮 → 前端 `POST /api/search`，期间检索结果区显示加载状态。
5. 收到响应后，按 `results` 顺序渲染结果卡片网格。
6. 用户可以重新输入文本再次检索，或更换参考视频。

### 3.3 结果卡片展示

每张卡片至少包含：

- 视频缩略图（点击可在弹窗中预览原视频，如有静态预览即可）
- 相似度分数（建议以百分比或星级呈现，原始数值仅在调试模式可见）
- 视频时长

**当 `debug=true` 时**，卡片底部额外展示一个折叠区域：
- 三个分支的注意力分数（retain / inject / exclude），用横向条形图
- 预测的动作类别标签

调试模式入口建议放在页面右上角一个不显眼的开关，用于答辩演示。

### 3.4 边缘状态（必须处理）

| 状态 | 触发条件 | 展示方式 |
|---|---|---|
| 加载中 | 检索请求发出后 | 结果区显示骨架屏或加载动画 |
| 全部低置信 | 所有 `results` 的 `low_confidence` 均为 `true` | 顶部出现提示："视频库中可能没有完全匹配的结果，以下是最接近的几条" |
| 部分低置信 | 仅部分结果 `low_confidence=true` | 这些卡片视觉降权（灰度或半透明） |
| 参考视频未选 | 用户点击检索时未选参考视频 | 检索按钮置灰，或弹出提示要求先选 |
| 修改文本为空 | 同上 | 同上 |
| 请求失败 | HTTP 非 2xx | 结果区显示错误提示，不清空已有结果 |

**重要：** 即使所有结果都低置信，也仍然展示它们（按后端返回顺序），只是加上整体提示。不要呈现"无结果"空状态。

### 3.5 不需要做的事

- ❌ 上传按钮、文件选择控件
- ❌ 用户登录、注册、个人中心
- ❌ 检索历史、收藏
- ❌ 视频上传进度条
- ❌ 国际化、暗色模式等非核心功能（除非进度允许再加）

## 4. 与 AI 模型团队的交付边界

| 项目 | 负责方 | 交付物 |
|---|---|---|
| 训练好的模型权重 | AI 团队 | checkpoint 文件 |
| 模型推理封装 | AI 团队 | Python 模块，导出 `encode_query` / `search` 两个函数 |
| 视频库 AIM 编码 | AI 团队 | 离线运行的索引脚本，产出 Faiss 索引 + id 映射 |
| 视频缩略图 | AI 团队（可商议） | 与索引同目录的静态图片，路径写入 metadata |
| 视频元数据表 | AI 团队 | JSON 或 SQLite，字段固定为 §2.1 表格中所列 |
| HTTP 服务、路由、参数校验 | 后端 | 按 §2 实现 |
| 前端页面、交互、视觉 | 前端 | 按 §3 实现 |

**约定不变项（一旦确定不要随意改）：**
- 向量维度 D = `<AI 团队填写>`
- 视频库规模 N = `<AI 团队填写>`
- 低置信阈值 THRESHOLD = 配置文件可调，初始 0.3
- `top_k` 默认 20，上限 50

## 5. 里程碑建议

| 阶段 | 后端 | 前端 | 验收方式 |
|---|---|---|---|
| M1 框架打通 | 假数据返回固定结果的 `/api/search` 和 `/api/videos` | 页面骨架、能调通假数据接口 | 前后端联调，能看到固定结果 |
| M2 接入模型 | 真实调用 `encode_query` / `search` | 完成结果卡片真实数据展示 | 用一个真实 query 跑通端到端 |
| M3 完善 UX | 实现低置信、参考过滤、错误处理 | 实现高级模式、调试视图、边缘状态 | 走一遍 §3.4 的所有边缘状态 |
| M4 演示打磨 | 调延迟、加日志 | 视觉优化、答辩演示脚本 | 模拟答辩演示一次 |

## 6. 答辩演示注意点

- 准备 2~3 个能凸显模型能力的样例（特别是包含动作变化的，例如"跑步 → 走路"、"打开 → 关闭"），用于演示 AIM 时序模块的效果。
- 演示时打开调试视图，让评审能看到三个分支分数和动作类别预测，说明模型不是"黑盒返回一个相似视频"。
- 准备一个低置信样例，演示前端如何提示"库中没有完全匹配的结果"，体现系统具有不确定性建模能力。
